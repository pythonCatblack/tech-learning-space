16.2 深度学习模型优化

16.2.1 模型量化

模型量化是嵌入式部署最重要的优化手段——将FP32模型转换为INT8或FP16，显著降低计算量和内存占用。

📉

FP32 → FP16

精度损失小，通常< 1%，推理加速约2倍。TensorRT和ONNX Runtime均支持。

📊

FP32 → INT8

精度损失稍大，需要校准数据集，加速可达4-8倍。主流目标检测模型均有成熟量化方案。

16.2.2 TensorRT加速

TensorRT是NVIDIA的推理优化框架——它通过算子融合、内核自动调优、内存优化等手段，可以将深度学习推理速度提升5-10倍。

🛠️ 动手实践

TensorRT INT8量化完整流程（YOLOv8）

# ① 导出ONNX模型（PyTorch）
python export.py --weights yolov8n.pt --include onnx

# ② 使用trtexec进行INT8量化
trtexec \
  --onnx=yolov8n.onnx \
  --int8 \
  --calib=./calib_dataset/ \
  --saveEngine=yolov8n_int8.engine \
  --verbose

# ③ 在Jetson Orin上运行
./yolov8n_int8.engine

# 性能对比
# FP32: 25 FPS @ Orin Nano
# FP16: 55 FPS @ Orin Nano  
# INT8: 120 FPS @ Orin Nano  ← 实际可满足实时避障要求

16.3 实时化与功耗优化

16.3.1 ROS2实时化

ROS2的实时性比ROS1大幅提升——支持确定性DDS通信、节点级别的实时线程调度。对于飞控链路（感知→决策→控制），实时性是硬性要求。

⚡

动态电压频率调节 (DVFS)

根据计算负载动态调整CPU/GPU频率和电压。轻载时降频省电，满载时超频保证性能。

🧊

模型层剪枝

去除冗余神经元和连接，减少计算量。常用方法：结构性剪枝（去除整个通道）。

🎯

Jetson电源模式

Orin Nano支持15W/25W/40W三档功率模式。在无人机上通常选MAXN 40W模式以保证算力。

16.3.2 功耗测试实战

🔑 无人机感知系统的功耗预算

以DJI Matrice 350 RTK为例：
• 起飞重量：约2.5kg
• 电池容量：5880mAh @ 26.1V ≈ 153Wh
• 悬停时间：约35分钟
• 感知系统功耗预算：10-20W（占总功耗约30%）

实测参考（Orin Nano + Livox Mid-70）：
• 待机（无负载）：5W
• LIO-SAM运行中：15-20W
• 峰值（LIO-SAM + 语义分割）：25-30W
• 剩余功率用于飞控、通信、电机 → 实际悬停时间会缩短到约30分钟。

16.4 工程实践清单

1

算法选型

根据平台算力选择算法复杂度（Orin Nano上运行完整LIO-SAM，Edge TPU上运行轻量级感知）。

2

模型量化

将深度学习模型量化到FP16/INT8，用校准数据集验证精度损失可接受。

3

算子融合

使用TensorRT或供应商SDK将多个小算子融合为一个，减少内存访问。

4

功耗测试

在实际飞行中测量感知系统的功耗，确保不超过飞行预算。

⚠️
常见陷阱① - INT8量化精度崩溃：某些模型（如小模型YOLOv8n）对INT8量化非常敏感——精度下降超过10%。解决：先用FP16验证精度，INT8仅在精度损失< 2%时才可接受。

常见陷阱② - 忽视散热导致降频：Orin Nano在25°C环境温度+密闭无人机机体内时，芯片温度可能超过85°C，触发热保护自动降频至15W模式——算力从40TOPS骤降至15TOPS，导致LIO-SAM丢帧。必须加散热片+主动风扇。

常见陷阱③ - CUDA内存泄漏：在长时间飞行中（如1小时+），点云处理如果不及时释放GPU内存，会导致显存碎片化，最终OOM崩溃。每帧处理完后必须显式调用 cudaFree。
